    2. Basisframework voor een Android-client die communiceert met een webservice / jSON

    We bieden nu een basisstructuur voor een Android-app die communiceert met een of meer webservices / jSON. Dit is het project [client-android-skel] dat te vinden is in de map [architecture] met voorbeelden:

      

    Het bestuderen van deze basisstructuur biedt de gelegenheid om bepaalde punten die we in de vorige voorbeelden zijn tegengekomen nog eens door te nemen. Deze applicatie zal als basis dienen voor alle toekomstige applicaties. Ze is na talrijke iteraties tot stand gekomen. Het doel is om zoveel mogelijk elementen van de applicaties die we binnenkort gaan bouwen in abstracte klassen onder te brengen, om te voorkomen dat we steeds hetzelfde soort code moeten schrijven dat alleen in details verschilt. De kenmerken zijn als volgt:

    • de asynchrone communicatie met de webserver / jSON verloopt via de bibliotheek RxJava;
    • de levenscyclus van een fragment (update, save, restore) wordt beheerd door de bovenliggende klasse [AbstractFragment], die op specifieke momenten bepaalde methoden van de onderliggende klassen aanroept. De onderliggende klasse hoeft zich dus geen zorgen te maken over de fasen van de levenscyclus, maar hoeft alleen bepaalde methoden te implementeren die door de bovenliggende klasse worden opgelegd;
    • de levenscyclus van de activiteit (opslaan / herstellen) wordt beheerd door een abstracte klasse [AbstractActivity], die op haar beurt de dochteractiviteit verplicht bepaalde methoden te implementeren;
    • de klasse [AbstractActivity] kan een applicatie beheren met of zonder tabbladen, met of zonder laadafbeelding, met of zonder basisauthenticatie bij de webserver / jSON. Of deze elementen al dan niet aanwezig zijn, wordt bepaald via de configuratie;

    Dit skelet is gebruikt voor alle volgende voorbeelden. Vanwege de diversiteit daarvan werkte wat voor het ene voorbeeld werkte, mogelijk niet voor het volgende. Aangezien het skelet voor in totaal zeven voorbeelden is gebruikt, hebben er talrijke iteraties plaatsgevonden. Als we het zouden gebruiken voor een achtste voorbeeld, is het mogelijk dat we opnieuw zouden merken dat de specifieke kenmerken van dit nieuwe voorbeeld nieuwe fouten veroorzaken. Niettemin zal het gebruik van dit skelet het schrijven van de komende voorbeelden aanzienlijk vereenvoudigen. Het beheer van de levenscyclus van een fragment (update, save, restore), in combinatie met het begrip ‘aangrenzende fragmenten’, is namelijk bijzonder complex. Hier is dit volledig verborgen in de klasse [AbstractFragment].

    2.1. Architectuur van de Android-client

    De voorgestelde Android-client is gebaseerd op de volgende architectuur:

    • de laag [DAO] implementeert een interface [IDao]. Deze communiceert met de webserver / jSON;
    • er is slechts één activiteit die tevens de interface [IDao] implementeert. De weergaven maken hier gebruik van om de server te bereiken;
    • de weergaven worden geïmplementeerd door fragmenten;

    2.5.2. De fragmentcontainer [MyPager]

      
    
package client.android.architecture;

import android.content.Context;
import android.support.v4.view.ViewPager;
import android.util.AttributeSet;
import android.view.MotionEvent;

public class MyPager extends ViewPager {

  // regelt het vegen
  private boolean isSwipeEnabled;
  // scrollbesturing
  private boolean isScrollingEnabled;

  // constructors
  public MyPager(Context context) {
    super(context);
  }

  public MyPager(Context context, AttributeSet attrs) {
    super(context, attrs);
  }

  // te herdefiniëren methoden voor het beheren van het vegen
  @Override
  public boolean onInterceptTouchEvent(MotionEvent event) {
    // swipen toegestaan?
    if (isSwipeEnabled) {
      return super.onInterceptTouchEvent(event);
    } else {
      return false;
    }
  }

  @Override
  public boolean onTouchEvent(MotionEvent event) {
    // swipen toegestaan?
    if (isSwipeEnabled) {
      return super.onTouchEvent(event);
    } else {
      return false;
    }
  }

  // scrollcontrole
  @Override
  public void setCurrentItem(int position){
    super.setCurrentItem(position,isScrollingEnabled);
  }

  // setters
  public void setSwipeEnabled(boolean isSwipeEnabled) {
    this.isSwipeEnabled = isSwipeEnabled;
  }

  public void setScrollingEnabled(boolean scrollingEnabled) {
    isScrollingEnabled = scrollingEnabled;
  }
}

    Deze klasse breidt de standaard Android-klasse [ViewPager] uitsluitend uit om het vegen (regel 11) en het scrollen (regel 13) tussen weergaven te beheren.

    • regels 26-43: de methoden die het swipen uitschakelen als dit is uitgeschakeld;
    • regels 46-49: herdefinitie van de methode [setCurrentItem] die dient om de weergegeven weergave te wijzigen. Als het scrollen is uitgeschakeld, vindt de weergavewisseling plaats zonder scrollen. Let op: de ontwikkelaar kan deze werkwijze omzeilen door de methode [setCurrentItem(int position, boolean smoothScrolling)] te gebruiken, waarmee hij kan aangeven welk scrollgedrag hij wenst;

import client.android.architecture.custom.CoreState;

public interface ISession {

    We introduceren de interface [ISession] om de aanwezigheid van bepaalde methoden in de sessie verplicht te stellen:

    • regels 7-10: het nummer van de laatst weergegeven weergave (fragment);
    • regels 12-15: de status van een bepaalde weergave;
    • regels 17-24: we introduceren het begrip 'actie in uitvoering'. Er zijn er vier (regel 17):
      • RESTORE: er vindt een opslag/herstel plaats. Er is geen wijziging van weergave;
      • NAVIGATION: er vindt een navigatie plaats. We noemen hier navigatie een weergavewisseling waarbij de nieuwe weergave kan worden hersteld op basis van de laatste status die tijdens de sessie is opgeslagen;
      • SUBMIT: we kennen het type [SUBMIT] toe aan een lopende actie wanneer er een weergavewisseling plaatsvindt en de nieuwe weergave afhankelijk is van de algemene status van de activiteit en niet alleen van haar eigen status. Soms is het moeilijk om onderscheid te maken tussen NAVIGATION en SUBMIT. In dat geval wordt het meest algemene geval, SUBMIT, gekozen;
      • NONE: waarde van de actie wanneer deze nog geen eerste waarde heeft gekregen;

    init();

    super(context, attrs, defStyleAttr);
    init();

    mTypeface = Typeface.createFromAsset(getContext().getAssets(), "fonts/Roboto-Bold.ttf");

    ViewGroup mainView = (ViewGroup) getChildAt(0);

      if (tabViewChild instanceof TextView) {
        ((TextView) tabViewChild).setTypeface(mTypeface, Typeface.NORMAL);

  private String user;
  // wachtwoord
  private String mdp;

  public ClientHttpResponse intercept(HttpRequest request, byte[] body, ClientHttpRequestExecution execution) throws IOException {
    // HTTP-headers van het onderschepte verzoek HTTP

  private int delay;

  protected <T> Observable<T> getResponse(final IRequest<T> request) {
    // logboek

      Log.d(String.format("%s", className), String.format("delay=%s", delay));

    return Observable.create(new Observable.OnSubscribe<T>() {

          T response = request.getResponse();

          if (isDebugEnabled) {

            if (response instanceof String) {
              log = (String) response;
            } else {
              log = mapper.writeValueAsString(response);
            }
            Log.d(className, String.format("response=%s sur thread [%s]", log, Thread.currentThread().getName()));

          // het antwoord wordt naar de waarnemer verzonden
          subscriber.onNext(response);
          // het einde van de observable wordt gemeld
          subscriber.onCompleted();

          if (isDebugEnabled) {

            } catch (JsonProcessingException e1) {
              Log.d(className, String.format("Erreur jSON imprévue"));
            }
          }
          // er wordt een uitzondering gegenereerd
          subscriber.onError(new DaoException(e, 100));

  private void showFragment(int i) {
    if (i < FRAGMENTS_COUNT && mViewPager.getCurrentItem() != i) {
      // geen navigatie bij het selecteren van een tabblad via de software
      session.setNavigationOnTabSelectionNeeded(false);
      // we maken de twee tabbladen opnieuw aan vanwege een probleem met het lettertype van de titels

      session.setNumFragment(i);
      // tabblad nr. 2 wordt geselecteerd met navigatie
      session.setNavigationOnTabSelectionNeeded(true);
      tabLayout.getTabAt(1).select();
    }

Als er niets wordt gedaan, vindt er navigatie naar een fragment plaats. Dit wordt voorkomen door de booleaanse waarde [navigationOnTabSelectionNeeded] in de sessie in te stellen op faux. Deze booleaanse waarde wordt automatisch teruggezet naar vrai door de klasse [AbstractFragment] wanneer een fragment zichtbaar wordt;
    • regel 46: het nummer van het weer te geven fragment wordt in de sessie opgeslagen;
    • regels 48-50: tabblad nr. 2 met navigatie wordt geselecteerd (regel 48). Dit activeert de procedure [onTabSelected], die:
      • het fragment weergeeft waarvan het nummer in de sessie is opgeslagen;
      • het nummer van het geselecteerde tabblad in de sessie opslaat;

    2.9. Conclusion

    Aan het einde van dit hoofdstuk beschikken we over een voorbeeldproject [client-android-skel] van een Android-client die communiceert met een webservice / jSON met de volgende kenmerken:

    • de asynchrone communicatie met de webserver / jSON verloopt via de bibliotheek RxJava;
    • de levenscyclus van een fragment (update, save, restore) wordt beheerd door de bovenliggende klasse [AbstractFragment], die op specifieke momenten bepaalde methoden van de onderliggende klassen aanroept. Het onderliggende fragment hoeft zich dus geen zorgen te maken over de fasen van de levenscyclus, maar hoeft alleen bepaalde methoden te implementeren die door de bovenliggende klasse worden opgelegd;
    • de levenscyclus van de activiteit (opslaan / herstellen) wordt beheerd door een abstracte klasse [AbstractActivity], die op haar beurt de dochteractiviteit verplicht bepaalde methoden te implementeren;
    • de klasse [AbstractActivity] kan een applicatie beheren met of zonder tabbladen, met of zonder laadbeeld, met of zonder basisauthenticatie bij de webserver / jSON. De aanwezigheid of afwezigheid van deze elementen wordt bepaald via de configuratie;
